# Media Library Implementation Guide

This document captures the **current state** of the media library integration in this CMS, what is already implemented, what is only partially prepared, operational notes, and a roadmap for future improvements.

It is intentionally detailed so it can serve as:

- a project status snapshot
- onboarding documentation for future developers
- a backlog for media-related enhancements
- a reference for disk/storage decisions

---

## 1) Executive summary

The project already includes a **working first integration** of Spatie Media Library with Filament for post cover images.

### What is working today

- `spatie/laravel-medialibrary` is installed.
- `filament/spatie-laravel-media-library-plugin` is installed.
- The `media` database table exists.
- `App\Models\Post` implements `HasMedia` and uses `InteractsWithMedia`.
- The `Post` model defines two media collections:
  - `cover` → single-file collection
  - `gallery` → multi-file collection with admin UI available only when `CMS_POST_GALLERY_ENABLED` is enabled
- The `Post` model defines two image conversions:
  - `thumb` (`400x300`)
  - `banner` (`1200x630`)
- The Filament `Post` resource includes:
  - an upload field for the `cover` collection
  - a table column that renders the `cover` image
- Uploads are explicitly sent to the `public` disk from the Filament form.
- Responsive images are enabled on the upload field.
- The app already has the standard Laravel `public` disk and symbolic-link setup needed to expose uploaded files.

### What is not finished yet

- No standalone “Media Library” resource/page exists in the admin.
- The `gallery` collection is now available as an opt-in admin feature, but remains disabled by default for the boilerplate.
- A conservative `config/media-library.php` file is now published in the app.
- No explicit cloud-disk strategy is implemented yet.
- No media metadata workflow exists yet (alt text, captions, focal point, credits, copyright, etc.).
- Media persistence is now covered at the model level, while UI-level upload flow remains intentionally light.
- No media governance policy exists yet (retention, cleanup, naming, private/public rules, CDN strategy, etc.).

## 2) Package and integration stack

The current media stack is built from these packages in `composer.json`:

- `spatie/laravel-medialibrary`
- `filament/spatie-laravel-media-library-plugin`

This gives us:

- model-based media attachments
- collection-based organization
- image conversions
- support for multiple filesystem disks
- Filament-native upload and table display components

### Why this is a strong base

This stack is a good fit for a CMS because it supports:

- attaching files to domain models instead of inventing a separate ad-hoc file table
- future migration to S3/object storage without rewriting the content model
- per-collection rules (single vs multiple files, conversions, visibility)
- good admin UX through Filament

## 3) Current implementation in the codebase

### 3.1 Model integration: `app/Models/Post.php`

`Post` is the only model currently integrated with the media library.

It already:

- implements `Spatie\MediaLibrary\HasMedia`
- uses `Spatie\MediaLibrary\InteractsWithMedia`
- registers media collections
- registers media conversions

### Registered collections

#### `cover`

Defined as:

- single-file collection
- intended for the main visual of a post

Practical effect:

- a post can only keep one current cover image
- uploading a new cover replaces the previous one in the collection
- this is appropriate for article/card/listing thumbnails

#### `gallery`

Defined as:

- multi-file collection

Practical effect:

- the backend model is ready for multiple media items per post
- the admin UI can expose this collection when `CMS_POST_GALLERY_ENABLED=true`
- the boilerplate keeps it hidden by default to avoid forcing a gallery workflow
- no frontend rendering contract exists yet for gallery media

So today `gallery` is **implemented but intentionally opt-in**.

### Registered conversions

#### `thumb`

Configured as:

- width: `400`
- height: `300`
- runs on `cover` and `gallery`

Likely intended uses:

- admin table thumbnails
- card grids
- small listing previews

#### `banner`

Configured as:

- width: `1200`
- height: `630`
- runs on `cover`

Likely intended uses:

- social sharing previews
- hero sections
- wide listing components

### Important current limitation

Although conversions are defined in `Post`, the current Filament table column does **not explicitly request a conversion**. The table uses `SpatieMediaLibraryImageColumn::make('cover')`, which means conversion usage should be treated as **not yet intentionally wired for admin display**.

That means:

- conversions may still be generated by the media package
- but the current admin list view is not clearly standardized on `thumb`
- a future refinement should choose the conversion explicitly for predictable performance and visual consistency

## 4) Current Filament admin implementation

### 4.1 Upload field in `app/Filament/Resources/PostResource.php`

The `Post` resource includes this media upload field:

- `SpatieMediaLibraryFileUpload::make('cover')`
- collection: `cover`
- type: image
- image editor enabled
- max size: `5120` KB
- disk: `public`
- responsive images enabled

### What this means in practice

Editors can currently:

- upload a cover image while creating/editing a post
- replace the existing cover image
- use Filament’s image editing UI before saving

### Current UX contract for editors

The admin UI currently suggests this intended usage:

- preferred cover image aspect is roughly `1200 × 630`
- cover image is a primary visual, not a multi-image asset set
- media is attached directly to the post record, not managed independently

### 4.2 Table rendering in `PostResource`

The list table uses:

- `SpatieMediaLibraryImageColumn::make('cover')`
- collection: `cover`
- a default fallback image: `asset('favicon.ico')`

This means:

- a post with a cover image shows an image in the table
- a post without a cover shows the fallback icon
- there is already a usable editorial view for cover-image presence

### Current gap in admin IA (information architecture)

The admin panel provider defines a **Media** navigation group in `app/Providers/Filament/AdminPanelProvider.php`, but there is **no dedicated media resource/page** living under that group yet.

So today:

- media exists as an attachment capability inside `Post`
- but there is no central media management area
- the `Media` navigation group is effectively a placeholder for future expansion

## 5) Database layer

### 5.1 `media` table exists

The migration `database/migrations/2026_04_14_040228_create_media_table.php` is present and creates the expected Spatie media table.

Stored attributes include:

- owning model (`morphs('model')`)
- `uuid`
- `collection_name`
- `name`
- `file_name`
- `mime_type`
- `disk`
- `conversions_disk`
- `size`
- `manipulations`
- `custom_properties`
- `generated_conversions`
- `responsive_images`
- `order_column`
- timestamps

### Why this matters

This schema already supports more than the current UI uses.

For example, the table is ready for future features such as:

- custom metadata via `custom_properties`
- explicit conversion-disk strategies
- ordering galleries
- tracking responsive image generation
- UUID-based media identifiers

### 5.2 Post table does not store file columns

The `posts` table does **not** include columns like:

- `cover_image`
- `image_path`
- `attachment_url`

That is good and intentional.

Media is modeled separately through the media library table instead of mixing file-path concerns into content columns.

This keeps the content model cleaner and makes future disk migration easier.

## 6) Filesystem and disk setup

### 6.1 Current configured disks

`config/filesystems.php` currently defines:

- `local`
- `public`
- `s3`

### 6.2 Current effective disk for media uploads

Even though the default filesystem disk is environment-driven via:

- `FILESYSTEM_DISK` → default fallback `local`

The current post cover upload explicitly uses:

- `disk('public')`

So the current real behavior for post cover images is:

- uploaded originals go to the `public` disk
- media is expected to be web-accessible through Laravel’s storage symlink

### 6.3 Public disk behavior today

The `public` disk is configured as:

- root: `storage/app/public`
- url: `APP_URL/storage`
- visibility: `public`

And the symbolic link mapping is already defined:

- `public/storage` → `storage/app/public`

### Operational implication

For media URLs to be accessible in browser contexts, the standard command should be run:

```bash
php artisan storage:link
```

This is already mentioned in `docs/filament-user-manual.md` under the “Files/media not visible” troubleshooting section.

### 6.4 `s3` disk exists but is not yet used by the media feature

The project already has a standard `s3` filesystem disk configuration scaffold.

However, the current media integration does **not** yet:

- switch media uploads to `s3`
- split originals and conversions across different disks
- document an object-storage strategy
- define CDN/public URL behavior for remote storage

So cloud storage readiness is **possible**, but not yet implemented as a project convention.

## 7) Current runtime flow

This is the current media lifecycle for a post cover image.

### Create/edit flow

1. An editor opens the post create or edit page.
2. The `cover` upload field accepts an image.
3. Filament sends the uploaded file through the media library plugin.
4. The file is attached to the `Post` model in the `cover` collection.
5. The media record is persisted in the `media` table.
6. The physical file is stored on the `public` disk.
7. Responsive image data may be generated because `responsiveImages()` is enabled on the field.
8. Model-defined conversions such as `thumb` and `banner` are available to the media system.

### List flow

1. The posts table loads records.
2. The image column attempts to resolve media from the `cover` collection.
3. If media exists, an image is rendered.
4. If not, the favicon fallback is rendered.

## 8) What has been implemented well already

The current implementation is already good in several ways.

### Strong points

#### 1. Media is attached to the domain model correctly

The project uses model-attached media rather than stuffing file paths into the `posts` table.

That is the right long-term shape for a CMS.

#### 2. The `cover` collection is constrained properly

Using `singleFile()` for a cover image is a good editorial rule.

#### 3. Conversions are declared in the model

This centralizes image derivation logic in one place instead of scattering width/height assumptions across views.

#### 4. Admin UX is already functional

Editors can upload and preview a cover image without needing a separate media module.

#### 5. Storage is explicit

The upload field explicitly targets the `public` disk, which reduces ambiguity.

#### 6. The schema supports future growth

The presence of `gallery`, `custom_properties`, `conversions_disk`, and `responsive_images` means the foundation is broader than the initial UI.

## 9) Current limitations and gaps

This section is important because the media feature is **partially implemented**, not “complete”.

### 9.1 Only `Post` is media-enabled

Right now, only `App\Models\Post` participates in media handling.

Potential future candidates:

- `Category` thumbnails/banners
- author avatars
- site-wide assets (logos, OG images, favicon variants)
- downloadable attachments
- page-builder assets

### 9.2 Only one collection is actually exposed in the UI

`cover` is the only usable collection in the current admin.

`gallery` is defined but not exposed.

### 9.3 No dedicated media management screen

There is no centralized experience for:

- browsing all uploaded files
- finding unused/orphaned assets
- renaming media
- editing metadata
- reusing a previously uploaded image across records
- bulk deleting media

### 9.4 No metadata conventions yet

There is currently no documented or implemented handling for:

- alt text
- image title
- caption
- copyright/license
- attribution/source
- focal point / crop intent
- photographer/creator

This means the current implementation is sufficient for raw image attachment, but not yet for mature editorial workflows.

### 9.5 No explicit conversion-display convention

Conversions exist, but the current list-table image display does not clearly standardize which conversion should be used.

That matters for:

- performance
- consistency
- frontend expectations
- cache predictability

### 9.6 Media config is now published, but intentionally minimal

The project now includes a committed `config/media-library.php` file.

That means:

- the current media defaults are explicit in application code
- maintainers can see the intended disk and queue behavior without reading vendor config
- projects can override media behavior through environment variables without editing source code

The config is intentionally conservative and does **not** try to expose every package option yet.
That keeps the boilerplate customizable without locking projects into premature decisions.

### 9.7 Media coverage now exists, but can grow further

Current tests now cover real model-level media persistence behavior.

Covered now:

- attaching a cover image to a post
- asserting media records exist in the `media` table
- asserting files land on the expected disk
- asserting `singleFile()` replaces older cover media
- asserting the `gallery` collection accepts multiple items
- asserting the post form reads its media disk from config
- asserting the gallery field is hidden by default and visible only behind a feature flag

Still not covered yet:

- browser-level upload/editor interaction
- conversion generation details
- responsive image generation details
- dedicated gallery reordering behavior through the admin UI

### 9.8 No seed/demo media

The current `CmsDemoSeeder` creates categories and posts, but does not attach media.

That means:

- demo environments may not showcase the media feature well
- empty-image states are common after seeding
- onboarding is slightly weaker than it could be

### 9.9 No cleanup or lifecycle policy

There is no documented answer yet for questions like:

- what happens to media when a post is deleted?
- how should orphaned files be audited?
- do we need scheduled cleanup commands?
- should revisions keep historic assets?
- do we retain originals forever?

These are not necessarily bugs, but they are important operational decisions for a production CMS.

## 11) Disk integration strategy: current state and future options

This is one of the most important sections for future-proofing.

## 11.1 Current disk strategy

Current effective policy:

- originals uploaded from the post form go to the disk defined in `config/media-library.php`
- the boilerplate default is `public`
- no explicit separate conversions disk is configured
- no private disk strategy is documented
- no CDN layer is documented

This is completely reasonable for the current stub/MVP stage.

## 11.2 Recommended near-term disk policy

For a simple CMS deployment, a good next step is:

- keep images on `public` for local/dev environments
- optionally move to `s3` for staging/production
- document one standard way to generate public URLs
- explicitly publish and commit media config

## 11.3 Possible future storage topologies

### Option A — All media on local public disk

Best for:

- local development
- very small single-server deployments
- fastest initial setup

Pros:

- simple
- cheap
- minimal moving parts

Cons:

- harder to scale horizontally
- backups matter more
- CDN integration is manual

### Option B — Originals and conversions both on S3

Best for:

- multi-server deployments
- scalable production infrastructure
- CDN-friendly delivery

Pros:

- durable object storage
- easy CDN fronting
- simpler shared-storage story

Cons:

- more env setup
- remote I/O costs and latency considerations

### Option C — Originals on private/object storage, conversions on public/CDN disk

Best for:

- controlled original assets
- performance-focused public delivery
- teams with stronger infrastructure maturity

Pros:

- clearer separation of source vs derived assets
- can restrict access to originals
- strong CDN story

Cons:

- more operational complexity
- must explicitly standardize URL generation and invalidation

## 11.4 Things to decide before switching disks

Before formalizing disk strategy, answer these questions:

1. Are all current media assets public by design?
2. Will the CMS eventually support private documents or member-only downloads?
3. Should originals be publicly accessible, or only conversions?
4. Will a CDN be used?
5. Do we need region-specific object storage?
6. Should conversions be generated synchronously or via queue in production?
7. Do we need temporary URLs for private assets?

## 11.5 Strong recommendation for this project

A practical progression for this codebase would be:

### Phase 1

- keep `public` disk for local development
- publish media config
- document media env variables
- add upload tests

### Phase 2

- support `s3` for non-local environments
- explicitly configure media disk selection
- explicitly configure conversion disk behavior
- decide on public URL and CDN rules

### Phase 3

- introduce private/public collection taxonomy
- add signed/temporary URL support where needed
- add cleanup/audit operations

### Recommended conventions

#### For `Post`

- `cover` → primary hero/listing image, single file
- `gallery` → supporting images, multiple files
- `attachments` → downloadable files (PDF, DOCX, ZIP), future
- `social` → optional custom OG images, future

#### For `Category` (future)

- `thumbnail`
- `banner`

#### For site-wide settings (future)

- `site-logo`
- `site-logo-dark`
- `favicon`
- `default-og-image`

### Why this matters

Stable collection names become part of the app’s content contract.

That helps with:

- frontend expectations
- template reuse
- migrations
- imports/exports
- documentation

## 14) SEO, accessibility, and performance considerations

Media handling is not just storage; it affects content quality and page performance.

### SEO/accessibility

Future media workflow should support:

- required alt text for content-critical images
- decorative image marking where appropriate
- OG/social image conventions
- filename normalization and predictable naming

### Performance

Future work should standardize:

- explicit thumbnail conversion usage in admin tables
- responsive image rendering strategy in frontend templates
- lazy loading policy
- image format strategy (future WebP/AVIF discussion if relevant)
- max dimensions and upload-size limits per collection

### Governance

Eventually define:

- who can upload/delete media
- who can reuse media from existing records
- whether editors can replace covers after publication
- whether archived posts retain media indefinitely

## 16) Future ideas

These are not requirements yet, but they are good directions for a production-grade CMS.

### A. Central media manager

A Filament resource/page that allows:

- searching media by type/model/collection
- reusing an existing asset
- managing metadata
- finding unused media
- bulk delete or bulk move

### B. Reusable media picker

Instead of upload-only workflows, allow editors to:

- choose from previously uploaded media
- upload new media inline when needed
- standardize shared assets across content

### C. Background processing strategy

If image processing becomes expensive:

- move conversions to queues
- surface conversion status in admin if necessary
- retry failed processing jobs

### D. Media policies and permissions

Define roles like:

- upload media
- edit metadata
- delete media
- manage global assets

### E. Frontend delivery service abstraction

Wrap media URL generation behind app-level helpers/services so future changes in:

- disk
- CDN domain
- image format
- signed URLs

can happen without touching every template.

### F. Import pipeline

If this CMS will ingest legacy content, add support for:

- remote URL download
- duplicate detection
- metadata normalization
- collection mapping rules
